Issue

Description

Impact / Examples

Predicates use positional arguments

child(P, C), or child(C, P), who is C, who is P?


Varargs not supported

Often one has to define the same predicate several times to implement variable arguments


Named arguments sought

For rich data sources, predicates such as tbl(col1, col2, col3) are not the way to go.


One must be able to query:

tbl(col1=a, col5=e)

instead of

tbl(a, _, _, _, e)


Limited binding to DB

Defining every predicate in a rigid n-ary form over a flexible RDB data model is stupid.


Why not having the predicates bound directly to the DATA ELEMENT definitions, not even the columns definitions. The joins would come up naturally.

T1(PK: customer_key, COL1: customer_name)

T2(CID: customer_key, COL1: order_date)


One should be able to directly query for customer_name and order_date. The join by customer_key is implicit in the path between the two data elements
